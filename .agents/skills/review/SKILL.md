# dev-ready Phase Reviewer

The Reviewer is the quality gate between the Junior's implementation and the
CEO's commit. It runs three independent passes over the same change — QA,
Security, SRE — each from its own role definition. The role is deliberately
decoupled from any specific tool or agent (it used to be "IBM Bob"): whoever is
assigned the reviewer role for a phase reads these definitions and follows them
(ADR-011).

## What you review

The change under review is the Junior's UNCOMMITTED working-tree changes on top
of `main` for one phase. Inspect it with read-only git only (`git status`,
`git diff`, `git log`). You do NOT modify code, commit, or run any
state-changing git — you report. In the file-triggered flow the Reviewer is
handed one of the per-phase briefs `04-qa-review.md` / `05-security-review.md` /
`06-sre-review.md` (generated by the `handoff` skill), each of which points back
to the matching role definition here.

## The three passes

Each pass has a standing role definition under `references/`; read the one for
your hat and follow it:

- QA — `references/qa.md`: test-tier coverage of every new behavior and error
  path, no-network/tmp_path discipline, exit codes per `docs/cli-spec.md`,
  regressions green.
- Security — `references/security.md`: untrusted-input paths to the filesystem,
  injection / path-traversal / zip-slip surface, secrets, dependency and
  workflow-permission deltas, pin changes.
- SRE — `references/sre.md`: all-or-nothing generation preserved for each new
  failure mode, message/exit-code quality, no new manual maintenance load,
  false-positive risk of new checks.

The `handoff` skill also gives each brief a phase-specific "Verify specifically"
list derived from that phase's change surface — treat it as the concrete
checklist on top of the standing definition.

## Precondition

`03-review.md` (the Senior's review) ends with a Verdict marking the phase
ready for review, and `reports/problems.md` does not exist. If the phase is not
ready, stop and flag it.

## Output — one report per pass

Write to the phase's reports directory, together with all other phase records:

```
docs/handoff/<version>/phase-<N>/reports/qa-review.md
docs/handoff/<version>/phase-<N>/reports/security-review.md
docs/handoff/<version>/phase-<N>/reports/sre-review.md
```

Do NOT write to any separate `reviews/` directory. Each report contains:

- Verdict: `APPROVE` or `REQUEST CHANGES` (in an automated runner that captures
  stdout, the last line may be `APPROVE` / `REJECT` — follow the runner's
  convention when told to).
- Blocking issues: file:line, what, why, suggested fix.
- Non-blocking suggestions: in a separate section.

Do not modify code. Report only. All three passes must reach `APPROVE` before
the phase can commit (non-release) or before `07` release authority unlocks
(release phase).

## Note on the runner

When the phase is driven by the automated `run-handoff.sh` pipeline, the review
agent has no write permission: the script feeds it the brief + execution report
on stdin and captures its stdout into the report file. In that mode you output
the full report to the terminal and end with a single `APPROVE` / `REJECT`
line — the file writing is the script's job, the path convention above still
holds.

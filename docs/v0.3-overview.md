# dev-ready v0.3 Overview — Phases 1–3

Status (2026-07-21): The v0.3 code (covering FR-11 through FR-14, including pinned tool integrations and item-level component selection) is fully implemented, reviewed, and sits uncommitted in the working tree. Phase 3 provides this documentation close-out. The entire codebase will ship together in the release step as **v0.3.0** (the current version in `pyproject.toml` remains `0.2.2`; the version bump will happen as part of the release protocol in `07-release.md`, not during this docs close-out).

## What v0.3 adds

While dev-ready v0.2 introduced pruning of upstream repository maintenance files, project-specific README generation, verification leak guards, and the agent-team overlay, v0.3 adds item-level selection granularities and pinned tool integrations:
- **FR-11**: Generation stamp (`.dev-ready.json`) recording metadata, selected components/items, and upstream commit pin.
- **FR-12**: Pinned codebase-memory MCP server integration (`code-memory`).
- **FR-13**: Pinned `react-doctor` frontend tool integration and wrapper skill (`react-doctor`).
- **FR-14**: Granular item-level component selection for skills and MCP servers.

---

## FR-11 — Generation Stamp (`.dev-ready.json`)

Every project generated by dev-ready now includes a `.dev-ready.json` file in its root directory:
- **Format & Version**: Uses `stamp_version: 2` (v1 was introduced in Phase 1; Phase 2 bumped it to `2` when per-item pins were added).
- **Metadata Recorded**: Captures `dev_ready_version`, selected components and item IDs (each paired with its pin version or `null`), and the upstream repository URL and pinned commit hash.
- **Verification Integrity**: `verify_project` in `src/dev_ready/verify.py` treats `.dev-ready.json` as a required file. Leak-guard checks continue to ensure generator machinery like `.copier/` and `.copier-answers.yml` remain in `FORBIDDEN_PATHS` and are pruned.

## FR-12 — Codebase-Memory MCP Item (ADR-008)

Integrates the `codebase-memory-mcp` server into generated projects under pinned-dependency mode:
- **Catalog Item**: Identified as `code-memory` within the `mcp` component catalog in `src/dev_ready/manifest.json`.
- **Configuration**: When selected, dev-ready injects an MCP server entry into `.mcp.json` configuring `uvx codebase-memory-mcp==0.9.0`.
- **Pin Management**: The pin version (`0.9.0`) resides solely in `manifest.json` as data and is recorded in the project's `.dev-ready.json` stamp.

---

## FR-13 — React-Doctor Integration (ADR-009)

Integrates frontend diagnostic tools for React projects via two complementary mechanisms:
- **Package & Script Injection**: Injects `react-doctor@0.8.1` into the generated `frontend/package.json` under `devDependencies` and adds a `"doctor": "react-doctor"` entry to `scripts`. Injection uses robust JSON parse-modify-write logic rather than regex splicing.
- **Wrapper Skill**: Installs an original wrapper skill under catalog item ID `react-doctor` in `.claude/skills/react-doctor/`. The skill guides the agent on when and how to run `npm run doctor` and interpret its findings, without redistributing third-party source code.
- **Pin Management**: The `0.8.1` pin version is declared in `src/dev_ready/manifest.json` and recorded in `.dev-ready.json`.

---

## FR-14 — Item-Level Component Selection (ADR-010)

Replaces all-or-nothing component selection with a flexible two-level selection architecture:
- **Two-Level Selection Flow**: Users can select level-1 components (`skills`, `mcp`, `docs`, `agents`) and level-2 items inside `skills` (e.g. `project-orientation`, `react-doctor`) and `mcp` (e.g. `mcp-config`, `code-memory`).
- **Defaults & Backward Compatibility**: All items are enabled by default. Accepting default interactive prompts or passing `--yes` reproduces pre-v0.3 output byte-identically.
- **CLI Options**: Adds `--skills <ids|all|none>` and `--mcp <ids|all|none>` flags. The `--no-skills` and `--no-mcp` flags act as aliases for `--skills none` and `--mcp none`. Passing invalid item IDs causes fast failure with exit code `2` listing valid IDs.
- **Data-Driven Catalog**: All component items, paths, pins, and injection metadata are defined declaratively in `src/dev_ready/manifest.json`.

## Release Gate for v0.3.0

To release v0.3 as version `v0.3.0`, the following sequence must pass:

1. **Unit Test Suite**: `uv run pytest` passes 100%.
2. **Lint Verification**: `uv run ruff check .` reports zero errors.
3. **Network Integration**: `uv run pytest -m network` passes.
4. **CI Gate**: GitHub Actions `generate-and-verify` workflow succeeds.
5. **Multi-Agent Review**: QA, Security, and SRE review passes approve the release.
6. **Release Execution (`07-release.md`)**:
   - Bump version from `0.2.2` to `0.3.0` in `pyproject.toml` and `src/dev_ready/__init__.py`.
   - Update lockfile (`uv lock`).
   - Commit all phase changes with designated Conventional Commit messages.
   - Tag and publish release `v0.3.0`.

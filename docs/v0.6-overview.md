# v0.6 Overview — dev-ready

Released: v0.6.0 | Status: Released
Scope: FR-21, FR-22

## What v0.6 shipped

dev-ready v0.6 adds two lifecycle commands for projects that have already been
generated. Both operate from the project's `.dev-ready.json` stamp (FR-11) and
the running CLI manifest, without network access. The CLI remains argparse-based;
standard reports are plain text, with an optional JSON report for `check`. This
release adds no runtime or development dependencies.

## Phase 1 — `dev-ready check` (FR-21)

- Added offline, read-only drift inspection in `check.py`, which explicitly must
  not import `fetch`, perform network I/O, or modify the target project
  (`src/dev_ready/check.py:1-8`).
- `check` compares the recorded dev-ready version and upstream pin with the
  running manifest, validates required and forbidden paths, and checks selected
  item pins, paths, and injection effects (`src/dev_ready/check.py:48-175`).
- Exit code 0 means clean; 6 means a missing or invalid stamp (including a
  pre-v0.3 project); and 7 reports drift. `--json` provides the same report for
  CI (`src/dev_ready/cli.py:75-89`, `src/dev_ready/check.py:158-193`).
- Read-only behavior is enforced by `test_check_read_only_assertion` in
  `tests/unit/test_check.py`, which compares byte-identical project-tree
  snapshots after clean, JSON, and drift inspections.

## Phase 2 — `dev-ready upgrade` (FR-22)

- Introduced `stamp_version` 3, adding `project_name` and a SHA-256 inventory of
  managed files (`src/dev_ready/overlay/__init__.py:49-61,112-116`; parsed in
  `src/dev_ready/stamp.py:47-48,136-152`).
- `upgrade` re-applies only overlay-managed whole files using the stamp's
  recorded item selection. It never touches upstream application code or shared
  injection targets (`src/dev_ready/upgrade.py:107-172`).
- The conflict rule is conservative: a file whose current hash matches the
  recorded inventory may be replaced; a user-modified file is skipped and
  reported, never silently overwritten (`src/dev_ready/upgrade.py:143-163`).
- Planned writes are staged with backups and rolled back on failure; `--dry-run`
  previews the report without writing (`src/dev_ready/upgrade.py:189-244`).
  Version-1 and version-2 stamps remain checkable, but upgrade refuses them with
  an explanation and exit code 8 (`src/dev_ready/upgrade.py:89-93`); the other
  upgrade outcomes are 0, 6, and 9.

## Notable decisions

- No new dependencies: lifecycle commands use the existing standard-library and
  package facilities.
- Module boundaries hold: `check`, `stamp`, and `upgrade` are offline and do not
  import `fetch`.
- All-or-nothing behavior now covers `upgrade` as well as initial generation.
- Read-only `check` and overlay-only `upgrade` behavior are enforced by tests,
  including byte-identical tree assertions, rather than convention alone.

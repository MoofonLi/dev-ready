# dev-ready v0.1 Overview — Phases 1–6

Status (2026-07-14): Phases 1–5 are released as v0.1.0–v0.1.2 on PyPI (v0.1.0/v0.1.1 are placeholders slated for yanking). Phase 6 (Copier consumption + release-pipeline hardening) is implemented and sits uncommitted in the working tree, targeting **v0.1.3**. Phase 6 reviews: QA APPROVE; Security and SRE REQUEST CHANGES — two small blocking fixes remain before release (see "Release gate for v0.1.3").

## What this project is

dev-ready is a Python CLI (`uvx dev-ready init <name>`) that generates an "AI-assisted-development-ready" FastAPI full-stack project in a single command: the base is `fastapi/full-stack-fastapi-template` consumed via Copier at a manifest-pinned commit, with dev-ready's own overlay on top (CLAUDE.md, Claude Code skills, MCP configuration, design-doc templates).

Two core architecture decisions run through all phases:

- **ADR-001/002 — No fork, pin commits**: upstream is consumed at a commit pinned in `src/dev_ready/manifest.json` (a lockfile shipped inside the wheel). Never fetch "latest" at generation time. Upstream updates come from a weekly CI job that opens bump PRs, fully validated by CI and merged manually.
- **All-or-nothing**: the entire pipeline runs in a staging directory; only the final step moves it into the target. If any step fails, the target is never touched and no half-finished output is left behind.

Since Phase 6 (ADR-005), fetching means running Copier — the template's officially supported consumption path — which also answers the template's own questions: the user's project name, a slugified stack name, and per-project generated secrets land in the generated project's `.env` (no more upstream `changethis` placeholders).

## End-to-end flow

```
uvx dev-ready init my-app
        |
        v
+------------------+   +------------------+   +------------------+
| 1. prompts       |   | 2. fetch         |   | 3. overlay       |
|  interactively    |-->|  copier run_copy  |-->|  apply CLAUDE.md/|
|  collect missing  |   |  @ pinned commit |   |  skills/MCP/docs |
|  answers + confirm|   |  answers+secrets |   |  template variable|
|  (--yes skips all)|   |  -> .env, staging|   |  substitution    |
+------------------+   +------------------+   +------------------+
                                                       |
        +----------------------------------------------+
        v
+------------------+   +------------------+   +------------------+
| 4. verify        |   | 5. finalize      |   | 6. report        |
|  structural checks|-->|  move staging to |-->|  success summary |
|  (6 required     |   |  target (the only|   |  + next steps    |
|  paths)          |   |  step touching   |   |                  |
|  fail = exit 5   |   |  target)         |   |                  |
+------------------+   +------------------+   +------------------+
```

Execution-order guarantees: the confirmation summary happens before any network call or write; in non-TTY environments, missing input exits 2 immediately and never hangs; steps 2–4 all happen inside a `tempfile.mkdtemp()` staging directory, with cleanup guaranteed by `finally`.

Exit codes (`docs/cli-spec.md`): 0 success; 1 unexpected error or user abort; 2 argument error; 3 network/fetch failure (including missing git and Copier errors since Phase 6); 4 target directory conflict; 5 generated output failed structural verification.

## Module map

| Module / file | Responsibility | Forbidden |
|---|---|---|
| `src/dev_ready/cli.py` | argparse, command dispatch, exception → exit-code mapping | No generation logic |
| `src/dev_ready/prompts/` | Interactive/non-interactive answer collection, unified into the `Answers` model (ADR-004) | I/O other than the terminal |
| `src/dev_ready/manifest/` | Load/validate manifest.json, single source of truth for pins | Being bypassed by other modules |
| `src/dev_ready/fetch/` | Run Copier against the pinned upstream commit into staging (ADR-005) | Knowing overlay contents; **the only module in the repo allowed to touch the network** |
| `src/dev_ready/overlay/` | Apply templates/ assets onto staging, template variable substitution | Any network call |
| `src/dev_ready/verify.py` | Structural post-gen checks (read-only) | Network, writes |
| `src/dev_ready/report/` | Pure-function rendering of the success summary | Any I/O |
| `src/dev_ready/generate.py` | Chains fetch→overlay→verify→move; builds the template answers (secrets) | — |
| `src/dev_ready/errors.py` | Typed exception hierarchy (each carrying its exit_code) | — |
| `src/dev_ready/templates/` | Overlay assets (CLAUDE.md.tmpl, skills, mcp.json, docs templates), accessed via `importlib.resources` | — |
| `scripts/bump_upstream.py` | CI-only pin bump tool (stdlib-only, not shipped in the wheel) | — |

Runtime dependencies: `questionary>=2.0,<3` (importable only from `prompts/`, lazily imported, never on the `--yes` path) and, since Phase 6, `copier>=9.4,<10` (lazily imported inside `fetch`, so `--help`/`--version` stay fast). git is now a host requirement (Copier clones; missing git = clean exit 3).

## What each phase delivered

### Phase 1 — Manifest loading + CLI skeleton (`8f4cb0e`, `c602b5d`)

- `src/dev_ready/manifest/` (frozen dataclasses `UpstreamPin`/`Manifest` in models.py; strict validation in loader.py: repo format, 40-hex commit).
- `src/dev_ready/cli.py` skeleton, `errors.py` exception hierarchy, `__main__.py`.
- `src/dev_ready/manifest.json` pinned to `fastapi/full-stack-fastapi-template@4cd0d9e5`.
- Lesson learned: the manifest directory was once swallowed by the gitignore `MANIFEST` rule; a fix commit followed.

### Phase 2 — Upstream snapshot fetching (`867b5a0`, follow-up fix `9bf9cd6`)

- `src/dev_ready/fetch/`: urls.py (codeload tarball URL, built only from validated pin fields), download.py, extract.py (safe extraction guarding against path traversal / symlinks / hardlinks), snapshot.py (fetch into temp, move only on success).
- Fetch-level all-or-nothing and exit 3.
- Review fixes: hardlink test, error messages carrying the URL, next-step guidance, known-limitation notes.
- (The tarball machinery built here was replaced by Copier in Phase 6; the staging/all-or-nothing pattern it established survives unchanged.)

### Phase 3 — Overlay + end-to-end wiring (`20c5614`, follow-up fix `4e2aad9`)

- `src/dev_ready/overlay/`: applies templates/ assets, `{{project_name}}`-style variable substitution, leftover-marker guard, collision checks (including dangling symlinks).
- `src/dev_ready/generate.py`: pipeline atomicity lifted from "fetch level" to "whole pipeline" — fetch into staging → overlay in staging → move only at the end.
- Packaging fix: assets moved from the repo-root `templates/` into `src/dev_ready/templates/`, switched to `importlib.resources` so they can be found after wheel installation (the key to making `uvx` work).
- At this point `dev-ready init <name> --yes` was usable end to end.

### Phase 4 — Interactive prompts + confirmation + reporting (`f85308d`, follow-up fix `cd4cc02`)

- `src/dev_ready/prompts/`: questionary (the first runtime dependency) hidden behind an injectable `Asker` interface; project-name prompt (re-asks in the same regex loop), component multi-select (skipped if any `--no-*` flag is present), confirmation summary.
- Guarantees: non-TTY missing input exits 2 and never hangs; Ctrl-C / cancel / rejected confirmation uniformly print `aborted: nothing was written` + exit 1 without a traceback; the `--yes` path never imports questionary.
- `src/dev_ready/report/render.py`: pure-function success summary, printing the actual written path and runnable next steps.
- Carry-over fixes (`cd4cc02`): ADR-004 convergence tests, non-TTY message assertions, questionary `<3` upper bound, cmd.exe compatibility caveat added to cli-spec, pinned-commit note added to the `docker compose watch` string.

### Phase 5 — Post-gen verification + CI/CD automation (`1a3448c`, `2d7bbd6`, `6c04688`)

- `src/dev_ready/verify.py`: 6 required paths (backend, frontend, compose.yml, compose.override.yml, .env, LICENSE — derived from actually fetching the pinned commit, not guessed), inserted after overlay and before move; failure = `VerificationError` exit 5, target untouched. Design principle: heavy verification (docker build + health check) lives in CI; the user's machine never needs Docker.
- `.github/workflows/ci.yml`: the `generate-and-verify` job — actually runs `dev-ready init demo-app --yes`, docker compose build + up, polls `GET /api/v1/utils/health-check/` (:8000, confirmed against the actual snapshot), 120-second budget with 24 retries, dumps logs on failure, teardown with `if: always()`, workflow-level `permissions: contents: read` (this job executes upstream code and must not see any secrets).
- `.github/workflows/upstream-bump.yml` + `scripts/bump_upstream.py`: every Monday 06:00 UTC, resolve upstream's latest commit (40-hex validated), update the manifest, and open a PR via `peter-evans/create-pull-request` (SHA-pinned) to the fixed branch `chore/upstream-bump` (reruns update the same PR — no spam). Verification is not reimplemented — the bump PR triggering ci.yml is the gate (ADR-002).
- `.github/workflows/release.yml`: tag version vs pyproject version gate → `uv build` → wheel smoke test (install into a scratch venv and run `--version`) → PyPI trusted publishing (OIDC, no tokens).
- `.github/dependabot.yml` (pip + github-actions, weekly), `THIRD_PARTY_NOTICES.md`, `docs/releasing.md` (one-time PyPI Trusted Publisher setup + release runbook).

### Phase 6 — Copier consumption + release-pipeline hardening (targeting v0.1.3, uncommitted)

- **ADR-005 (amends ADR-003)**: `fetch` now runs `copier.run_copy(https://github.com/<repo>.git, staging, data=..., vcs_ref=<pinned commit>, defaults=True, unsafe=True, quiet=True)`. `download.py`/`extract.py`/`urls.py` and their tar-safety tests are deleted. `generate._template_data()` supplies the template answers: project name, slugified `stack_name`, and `secrets.token_urlsafe` values for SECRET_KEY / POSTGRES_PASSWORD / FIRST_SUPERUSER_PASSWORD, written into the generated `.env` by the template's own `_tasks`. Generated projects also gain `.copier/.copier-answers.yml` (enables `copier update`).
- Accepted trade-offs (documented in ADR-005): new dependency tree (jinja2/pydantic/plumbum via copier), git as a host requirement, and — the material one — `unsafe=True` executes the template's pinned `_tasks` script on the end user's machine at generation time. Containment argument (pin is a 40-hex SHA, HTTPS-only clone, pin changes only via CI-gated bump PRs) was traced end-to-end and passed security review.
- `release.yml`: `environment: pypi` on the publish job (Deployments recording + PyPI trusted-publisher narrowing), `contents: write`, and a final `gh release create "$GITHUB_REF_NAME" --generate-notes dist/*` step so tags produce real GitHub Releases.
- READMEs: explicit Installation section (uvx / pip), git requirement, Copier wording.
- Process note: implemented out-of-band in a Cowork session rather than via the normal Architect → Engineer handoff; the deviation and its unverified claims were documented in `handoff/phase6/0-engineer-report.md`, then verified by Opus + Bob. Reviews: QA APPROVE (123 passed, ruff clean, real generation confirmed `.env` secrets); Security REQUEST CHANGES (1 blocking); SRE REQUEST CHANGES (2 blocking, 1 already resolved — `uv.lock` has been regenerated).

## Testing strategy and current state

Three test layers (`pyproject.toml` defaults to `-m 'not network'`, excluding network tests):

- **Unit (123, all run by default)**: no network, no filesystem outside tmp_path, no real TTY. Prompt behavior uses a fake asker; the bump script uses a `urlopen` monkeypatch; verify is parametrized over every entry in `REQUIRED_UPSTREAM_PATHS`; Copier is mocked in `test_fetch.py` (call contract: pinned `vcs_ref`, `defaults`/`unsafe`/URL, failure→FetchError, no temp leaks). Count dropped from 129: tar-safety tests left with the tarball code, copier-contract and template-data tests arrived.
- **Integration / E2E (2, `network` marker)**: `test_fetch_real.py`, `test_init_real.py` run a full init against real GitHub (now requiring git in the environment).
- **CI's generate-and-verify job**: the final line of defense for FR-5 — a real project + real Docker + real health check, required on every PR (including bump PRs).

## Development process (multi-agent workflow)

Each phase followed a fixed cycle: Architect (Claude, producing documents in `handoff/phaseN/`: engineer / QA / security / SRE) → Engineer (implements per 1-engineer and commits) → IBM Bob three-way review (written to `.bob/reviews/phaseN-*.md`) → Architect reconciles the reviews and schedules non-blocking items as the first carry-over commit of the next phase. Conventional Commits throughout (`feat:`/`fix:`/`ci:`/`docs:`), feeding the release automation. Phase 6 deviated (implementation preceded the handoff); the review gate caught real blockers, which is the system working as intended.

## Release gate for v0.1.3 (blocking, in order)

1. **Security blocker** — tighten `_REPO_PATTERN` in `src/dev_ready/manifest/loader.py` to GitHub's actual owner/name charset (see `.bob/reviews/phase6-security.md`, item 1). One-line change + its unit test.
2. **SRE B-1** — `docs/releasing.md` still instructs "Environment name: leave blank", contradicting `environment: pypi` in release.yml. Update the runbook: PyPI trusted publisher environment = `pypi`, plus a GitHub Environment pre-creation step (tag-pattern protection `v*`). (Note: per PyPI's documented semantics a blank publisher-side environment accepts tokens from any environment, so the review's day-1-failure scenario is likely overstated — but the runbook/workflow contradiction is real and the narrowing is wanted anyway.)
3. Bump `pyproject.toml` version to `0.1.3`, re-run `uv lock`, full suite + ruff green.
4. Yank v0.1.0/v0.1.1 placeholders on PyPI before installation docs circulate (SRE N-4; pip on Python 3.10/3.11 currently resolves to the broken 0.1.0 placeholder).
5. Commit (Conventional Commits split), push, confirm CI green, tag `v0.1.3`, push the tag.

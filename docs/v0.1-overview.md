# dev-ready v0.1 Overview — Phases 1–5

Status: v0.1 feature-complete (2026-07-14). All five phases landed; each phase passed IBM Bob's three-way QA / Security / SRE review (`.bob/reviews/`, all APPROVE, zero blocking issues). Not yet officially published to PyPI (see "Remaining work" at the end).

## What this project is

dev-ready is a Python CLI (`uvx dev-ready init <name>`) that generates an "AI-assisted-development-ready" FastAPI full-stack project in a single command: the base is a snapshot of `fastapi/full-stack-fastapi-template` at a manifest-pinned commit, with dev-ready's own overlay on top (CLAUDE.md, Claude Code skills, MCP configuration, design-doc templates).

Two core architecture decisions run through all five phases:

- **ADR-001/002 — No fork, pin commits**: upstream is fetched as a tarball snapshot, with the commit pinned in `src/dev_ready/manifest.json` (a lockfile shipped inside the wheel). Never fetch "latest" at generation time. Upstream updates come from a weekly CI job that opens bump PRs, fully validated by CI and merged manually.
- **All-or-nothing**: the entire pipeline runs in a staging directory; only the final step moves it into the target. If any step fails, the target is never touched and no half-finished output is left behind.

## End-to-end flow

```
uvx dev-ready init my-app
        |
        v
+------------------+   +------------------+   +------------------+
| 1. prompts       |   | 2. fetch         |   | 3. overlay       |
|  interactively    |-->|  codeload tarball |-->|  apply CLAUDE.md/|
|  collect missing  |   |  @ pinned commit |   |  skills/MCP/docs |
|  answers + confirm|   |  safe-extract to |   |  template variable|
|  (--yes skips all)|   |  staging         |   |  substitution    |
+------------------+   +------------------+   +------------------+
                                                       |
        +----------------------------------------------+
        v
+------------------+   +------------------+   +------------------+
| 4. verify        |   | 5. finalize      |   | 6. report        |
|  structural checks|-->|  move staging to |-->|  success summary |
|  (6 required     |   |  target (the only|   |  + next steps    |
|  paths)          |   |  step touching   |   |                  |
|  fail = exit 5   |   |  target)         |   |                  |
+------------------+   +------------------+   +------------------+
```

Execution-order guarantees: the confirmation summary happens before any network call or write; in non-TTY environments, missing input exits 2 immediately and never hangs; steps 2–4 all happen inside a `tempfile.mkdtemp()` staging directory, with cleanup guaranteed by `finally`.

Exit codes (`docs/cli-spec.md`): 0 success; 1 unexpected error or user abort; 2 argument error; 3 network/fetch failure; 4 target directory conflict; 5 generated output failed structural verification.

## Module map

| Module / file | Responsibility | Forbidden |
|---|---|---|
| `src/dev_ready/cli.py` | argparse, command dispatch, exception → exit-code mapping | No generation logic |
| `src/dev_ready/prompts/` | Interactive/non-interactive answer collection, unified into the `Answers` model (ADR-004) | I/O other than the terminal |
| `src/dev_ready/manifest/` | Load/validate manifest.json, single source of truth for pins | Being bypassed by other modules |
| `src/dev_ready/fetch/` | Download + verify + safe-extract upstream snapshots | Knowing overlay contents; **the only module in the repo allowed to touch the network** |
| `src/dev_ready/overlay/` | Apply templates/ assets onto staging, template variable substitution | Any network call |
| `src/dev_ready/verify.py` | Structural post-gen checks (read-only) | Network, writes |
| `src/dev_ready/report/` | Pure-function rendering of the success summary | Any I/O |
| `src/dev_ready/generate.py` | Chains fetch→overlay→verify→move into the all-or-nothing pipeline | — |
| `src/dev_ready/errors.py` | Typed exception hierarchy (each carrying its exit_code) | — |
| `src/dev_ready/templates/` | Overlay assets (CLAUDE.md.tmpl, skills, mcp.json, docs templates), accessed via `importlib.resources` | — |
| `scripts/bump_upstream.py` | CI-only pin bump tool (stdlib-only, not shipped in the wheel) | — |

The only runtime dependency: `questionary>=2.0,<3` (importable only from `prompts/`, lazily imported, never triggered on the `--yes` path).

## What each phase delivered

### Phase 1 — Manifest loading + CLI skeleton (`8f4cb0e`, `c602b5d`)

- `src/dev_ready/manifest/` (frozen dataclasses `UpstreamPin`/`Manifest` in models.py; strict validation in loader.py: repo format, 40-hex commit).
- `src/dev_ready/cli.py` skeleton, `errors.py` exception hierarchy, `__main__.py`.
- `src/dev_ready/manifest.json` pinned to `fastapi/full-stack-fastapi-template@4cd0d9e5`.
- Lesson learned: the manifest directory was once swallowed by the gitignore `MANIFEST` rule; a fix commit followed.

### Phase 2 — Upstream snapshot fetching (`867b5a0`, follow-up fix `9bf9cd6`)

- `src/dev_ready/fetch/`: urls.py (codeload tarball URL, built only from validated pin fields), download.py, extract.py (safe extraction guarding against path traversal / symlinks / hardlinks), snapshot.py (fetch into temp, move only on success).
- Fetch-level all-or-nothing and exit 3.
- Review fixes: hardlink test, error messages carrying the URL, next-step guidance, known-limitation notes.

### Phase 3 — Overlay + end-to-end wiring (`20c5614`, follow-up fix `4e2aad9`)

- `src/dev_ready/overlay/`: applies templates/ assets, `{{project_name}}`-style variable substitution, leftover-marker guard, collision checks (including dangling symlinks).
- `src/dev_ready/generate.py`: pipeline atomicity lifted from "fetch level" to "whole pipeline" — fetch into staging → overlay in staging → move only at the end.
- Packaging fix: assets moved from the repo-root `templates/` into `src/dev_ready/templates/`, switched to `importlib.resources` so they can be found after wheel installation (the key to making `uvx` work).
- At this point `dev-ready init <name> --yes` was usable end to end.

### Phase 4 — Interactive prompts + confirmation + reporting (`f85308d`, follow-up fix `cd4cc02`)

- `src/dev_ready/prompts/`: questionary (the first runtime dependency) hidden behind an injectable `Asker` interface; project-name prompt (re-asks in the same regex loop), component multi-select (skipped if any `--no-*` flag is present), confirmation summary.
- Guarantees: non-TTY missing input exits 2 and never hangs; Ctrl-C / cancel / rejected confirmation uniformly print `aborted: nothing was written` + exit 1 without a traceback; the `--yes` path never imports questionary.
- `src/dev_ready/report/render.py`: pure-function success summary, printing the actual written path and runnable next steps.
- Carry-over fixes (`cd4cc02`): ADR-004 convergence tests, non-TTY message assertions, questionary `<3` upper bound, cmd.exe compatibility caveat added to cli-spec, pinned-commit note added to the `docker compose watch` string.

### Phase 5 — Post-gen verification + CI/CD automation (`1a3448c`, `2d7bbd6`, `6c04688`)

- `src/dev_ready/verify.py`: 6 required paths (backend, frontend, compose.yml, compose.override.yml, .env, LICENSE — derived from actually fetching the pinned commit, not guessed), inserted after overlay and before move; failure = `VerificationError` exit 5, target untouched. Design principle: heavy verification (docker build + health check) lives in CI; the user's machine never needs Docker.
- `.github/workflows/ci.yml`: the `generate-and-verify` job — actually runs `dev-ready init demo-app --yes`, docker compose build + up, polls `GET /api/v1/utils/health-check/` (:8000, confirmed against the actual snapshot), 120-second budget with 24 retries, dumps logs on failure, teardown with `if: always()`, workflow-level `permissions: contents: read` (this job executes upstream code and must not see any secrets).
- `.github/workflows/upstream-bump.yml` + `scripts/bump_upstream.py`: every Monday 06:00 UTC, resolve upstream's latest commit (40-hex validated), update the manifest, and open a PR via `peter-evans/create-pull-request` (SHA-pinned) to the fixed branch `chore/upstream-bump` (reruns update the same PR — no spam). Verification is not reimplemented — the bump PR triggering ci.yml is the gate (ADR-002).
- `.github/workflows/release.yml`: tag version vs pyproject version gate → `uv build` → wheel smoke test (install into a scratch venv and run `--version`) → PyPI trusted publishing (OIDC, no tokens).
- `.github/dependabot.yml` (pip + github-actions, weekly), `THIRD_PARTY_NOTICES.md`, `docs/releasing.md` (one-time PyPI Trusted Publisher setup + release runbook).

## Testing strategy and current state

Three test layers (`pyproject.toml` defaults to `-m 'not network'`, excluding network tests):

- **Unit (129, all run by default)**: no network, no filesystem outside tmp_path, no real TTY. Prompt behavior uses a fake asker; the bump script uses a `urlopen` monkeypatch; verify is parametrized over every entry in `REQUIRED_UPSTREAM_PATHS`.
- **Integration / E2E (2, `network` marker)**: `test_fetch_real.py`, `test_init_real.py` run a full init against real GitHub.
- **CI's generate-and-verify job**: the final line of defense for FR-5 — a real project + real Docker + real health check, required on every PR (including bump PRs).

## Development process (multi-agent workflow)

Each phase followed a fixed cycle: Architect (Claude, producing four documents in `handoff/phaseN/`: engineer / QA / security / SRE) → Engineer (implements per 1-engineer and commits) → IBM Bob three-way review (written to `.bob/reviews/phaseN-*.md`) → Architect reconciles the reviews and schedules non-blocking items as the first carry-over commit of the next phase. Conventional Commits throughout (`feat:`/`fix:`/`ci:`/`docs:`), feeding the release automation.

## Remaining work

### Required manual steps before releasing v0.1 (maintainer-only)

1. **One-time PyPI Trusted Publisher setup** (full steps in `docs/releasing.md`): on PyPI, `dev-ready` project → Publishing → add the GitHub repo + `release.yml`. Without this, the release workflow fails at the publish step.
2. Release per the runbook: confirm the pyproject version → tag `vX.Y.Z` → push the tag.
3. Recommended: enable branch protection on the GitHub repo (require CI pass) so the bump-PR gate is actually enforced.
4. `.gitignore` has one uncommitted change (ignoring `handoff/` and `.bob/reviews/`) — if intentional, remember to commit it.

### Non-blocking backlog from reviews (none block the release)

- Pin `pypa/gh-action-pypi-publish` to a full SHA (raised by all of QA/SEC/SRE; dependabot will maintain it).
- Harden network isolation of the generate-and-verify job (the upstream container has full outbound network on the runner; blast radius is currently contained by zero secrets + read-only token).
- `verify_project` reports only the first missing path (could collect all of them).
- Explicit whitespace-sha test case for the bump script, per-docker-step timeouts, a Windows CI job (currently judged not worth adding).
- ANSI escape output for `--dir` (only affects the user's own terminal), UX semantics of `components_explicit`, driving the report's next-steps string from the manifest in the future (in the spirit of ADR-002).

### Planned, explicitly out of v0.1 scope (`docs/requirements.md` roadmap)

- `dev-ready check` (validate an existing project against the manifest) and `dev-ready upgrade` (re-apply a newer overlay).
- Support for other base templates; a Web UI companion (decision deferred).
- **Vendoring external skill content** (mattpocock/skills, the Apache 2.0 parts of anthropics/skills, caveman, awesome-design-md, etc.): currently everything in `templates/` is original content, and THIRD_PARTY_NOTICES records this accurately. Actually vendoring will require per-repo license handling (Apache NOTICE, excluding source-available and Commons Clause items) and provenance records in a manifest `vendored` section.
- v0.2: expanded overlay component-selection matrix, stronger non-interactive mode.

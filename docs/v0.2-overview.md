Phase 4 provides this documentation close-out. ## What v0.2 adds

While dev-ready v0.1 shipped the initial project generator and the interactive components matrix (skills, MCP config, and design-doc templates), v0.2 introduces a fourth optional overlay component (`agents`) along with three major generation-quality and reliability enhancements:
- **FR-7**: Pruning of upstream repo-maintenance and template-specific files.
- **FR-8**: Generation of a clean, project-specific root `README.md`.
- **FR-9**: Integration of a strict leak guard in the verification phase to prevent repository template leaks.
- **FR-10**: The agent-team overlay facilitating structured multi-agent coordination.

## FR-7 — Prune Upstream Repo-Maintenance Files (ADR-006)

To prevent upstream repository-specific metadata from leaking into generated user projects, dev-ready split how paths are omitted into two mechanisms:
- **`exclude`**: Files and directories that Copier never materializes in the staging directory in the first place. This includes `.git`, `copier.yml`, `copier.yaml`, and vendored skill directories.
- **`prune`**: A post-generation cleanup step. These are files that are valid output from Copier templates but are only meaningful for maintaining the upstream template repository.

Both lists are maintained strictly as data within `src/dev_ready/manifest.json` under `upstream.base_template`.

### Pruned Categories
- **Upstream Root Files**: `/README.md`, `CONTRIBUTING.md`, `release-notes.md`.
- **Media and Scripts**: `img/**`, `scripts/add_latest_release_date.py`, `hooks/**`.
- **GitHub Workflows and Labeling**: `.github/workflows/deploy-production.yml`, `.github/workflows/deploy-staging.yml`, `.github/workflows/issue-manager.yml`, `.github/workflows/labeler.yml`, `.github/workflows/add-to-project.yml`, `.github/workflows/latest-changes.yml`, `.github/workflows/smokeshow.yml`, `.github/workflows/detect-conflicts.yml`, `.github/workflows/zizmor.yml`, `.github/workflows/guard-dependencies.yml`, and `.github/labeler.yml`.

### Kept Files (The KEEP Set)
The application's own configuration and integration workflows are explicitly retained to ensure local development and CI continue to function. This "keep" set includes:
- GitHub Actions workflows: `test-backend.yml`, `test-docker-compose.yml`, `playwright.yml`, `pre-commit.yml`.
- General docs: `development.md`, `deployment.md`.
- Copier configuration folder: `.copier/`.

## FR-8 — Project-Specific README

Under dev-ready v0.1, the generated project included the upstream repository's `README.md`. In v0.2:
- The template overlay writes a new, clean `README.md` unconditionally (it is not gated by the `--no-docs` flag).
- The template README is dynamically populated with the generated project's name.
- This works safely because the post-generation prune step in **FR-7** deletes the upstream `README.md` first, ensuring that the overlay's "no-overwrite / no-collision" rule is preserved when writing the project-specific `README.md`.

## FR-9 — Leak Guard in Verify

To enforce quality and ensure that future upstream or Copier updates do not reintroduce repository metadata leaks, dev-ready has enhanced `verify_project` in `src/dev_ready/verify.py`:
- A list of `FORBIDDEN_PATHS` (`.git`, `copier.yml`, `copier.yaml`) is validated against the staging directory.
- If any forbidden paths exist in staging, generation fails loudly by raising a `VerificationError` and exiting with code `5`.
- This ensures that accidental leaks are blocked during testing and CI when bumping pins, long before they can impact end users.

## FR-10 — Agent-Team Overlay (ADR-007)

The `agents` component is introduced as a fourth optional component alongside `skills`, `mcp`, and `docs`.

### Features
- **Workflow Scaffold**: When enabled, the overlay writes to `docs/handoffs/`. This directory contains a `README.md` explaining the fixed-role, document-driven multi-agent coordination protocol, plus a set of templates under `phase-N/` (`01-opus-plan.md`, `02-gemini-implementation.md`, `03-opus-review.md`, `04-bob-qa.md`, `05-bob-security.md`, `06-bob-sre.md`).
- **CLAUDE.md Integration**: The generated root `CLAUDE.md` automatically carries a dedicated "Multi-agent handoff protocol" section pointing to `docs/handoffs/README.md`.
- **UI & CLI Integration**:
  - A `--no-agents` flag (type `bool`, defaulting to `false`) is added. By default, the overlay is enabled, and `--no-agents` disables it.
  - Interactive multi-select prompt flow displays `agents` as the fourth checkbox selection.
  - The confirmation summary includes `agents` in the active component list when selected.
  - See [cli-spec.md](cli-spec.md) for full interface details.

## Release Gate for v0.2.2

To release the cumulative v0.2 changes under version `v0.2.2`, the following sequence must be completed:

1. **Verification**: Confirm that the entire unit test suite passes (`uv run pytest`), ruff checks are clean (`uv run ruff check .`), and the integration tests (`uv run pytest -m network`) succeed.
2. **CI Check**: Verify that the GitHub Actions `generate-and-verify` job passes successfully.
3. **Review**: Ensure all QA, Security, and SRE checks pass (as verified by IBM Bob's checklists).
4. **Release Execution (Phase 7)**:
   - Bump version to `0.2.2` in `pyproject.toml` and `src/dev_ready/__init__.py`.
   - Update lockfile (`uv lock`).
   - Commit all uncommitted changes from Phase 3 and Phase 4 using their designated Conventional Commit messages.
   - Tag the release `v0.2.2`.

*Note: No code commits or version bumps occur during Phase 4.*

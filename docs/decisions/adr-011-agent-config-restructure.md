# ADR-011: Standard agent-config layout â€” AGENTS.md, docs/decisions/, docs/handoff/, .agents/skills/

- Status: Accepted (2026-07-20), amends ADR-007 (paths only; roles and protocol unchanged)
- Context: The repo's agent configuration grew organically: agent rules lived in a Claude-specific `CLAUDE.md` (invisible to Gemini/Antigravity, which reads `AGENTS.md` natively), ADRs were embedded in `docs/architecture.md` (hard to reference individually), handoffs sat in a root-level `handoff/` directory, and the process workflows (planning, handoff generation, release) existed only as skills in the Tech Lead's Cowork environment â€” unversioned and invisible to the repo. Meanwhile the ecosystem converged on standards: AGENTS.md (Linux Foundation open standard, read natively by Codex, Cursor, Copilot, Gemini CLI, Antigravity; Claude Code consumes it via `@AGENTS.md` import in CLAUDE.md), one-file-per-ADR under `docs/decisions/`, and the open Agent Skills format discovered at `.agents/skills/`.
- Decision: adopt the standard layout in four moves.
  - `AGENTS.md` at repo root is the single source of truth for agent rules; `CLAUDE.md` contains only `@AGENTS.md`.
  - ADRs move to `docs/decisions/adr-NNN-<slug>.md`, one file each; `docs/architecture.md` keeps the system overview, module boundaries, dependency rules, and an ADR index table.
  - The handoff tree moves to `docs/handoff/<version>/phase-N/` and REMAINS gitignored (CEO decision 2026-07-20): handoffs are regenerable working artifacts, not repo content; committing them would add template boilerplate noise to every phase. The full six-document protocol (plus `07-opus-release` for release phases) and `reports/` outputs are unchanged.
  - Process skills move into the repo at `.agents/skills/{planning,handoff,review,release}/SKILL.md` and become the single source of truth; the Tech Lead's Cowork-side skills are reduced to thin shells that read and follow the repo versions. `.claude/skills/` holds pointer stubs so Claude Code discovers them at its native path.
  - Agent identity is decoupled from role: no model name (Opus 4.8 / Gemini 3.1 Pro / Fable 5) appears anywhere in binding config â€” the role binds, the model is swappable and named at most once as a non-binding "current assignment" note. The review role (three passes: QA / Security / SRE) is likewise decoupled from any specific tool: its standing definitions move from `.bob/{qa,security,sre}.md` to `.agents/skills/review/references/{qa,security,sre}.md` under a new `review` skill, and the per-phase review docs are renamed `04-qa-review` / `05-security-review` / `06-sre-review` (previously `04-bob-qa` etc.). `.bob/` retains only the Bob-Shell notes mechanism the git hooks depend on; it is no longer the home of review role definitions.
- Consequences: Hard rules reach every agent from one file â€” the junior no longer depends solely on rules copied into `02-gemini-implementation.md` (the copies remain as reinforcement). Process changes are now versioned, reviewable commits instead of silent edits in one agent's environment. Individual ADRs are linkable from plans and handoffs. The review gate can be run by any capable agent, not only IBM Bob, without editing the workflow. Costs accepted: one-time path churn (`run-handoff.sh` REPO_ROOT + reviewer filenames, skill templates, AGENTS.md references) and a permanent duplication of skill discovery stubs under `.claude/skills/` until Claude Code reads `.agents/skills/` natively. Retained legacy (SUPERSEDED â€” see Amendment 2026-07-20): the `01-opus-plan` / `02-gemini-implementation` / `03-opus-review` / `07-opus-release` filenames keep their agent-label tokens as structural state-machine identifiers (the role, not the token, is binding); renaming them was judged higher-churn than its value. Historical handoff directories keep their old `phaseN` naming; new phases use `phase-N`.
